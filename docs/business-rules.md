# Бизнес-правила

## Подтверждено кодом

- В плане на одну дату хранится один выбранный ужин.
- Случайный подбор исключает неактивные, уже выбранные на неделе и содержащие запрещённые продукты блюда.
- Ручной выбор блюда не гарантирует отсутствие повторов; это ограничение интерфейса, а не подтверждённое правило.

## Повтор прошлой недели

- Неделя считается с понедельника по воскресенье в локальной timezone пользователя.
- Повторять можно только прошедшую непустую неделю; целевая неделя — текущая или будущая.
- По умолчанию заполняются только пустые дни. Замена конфликтов требует отдельного явного подтверждения.
- Копируются блюдо и соответствующий день недели; статус сбрасывается в `planned`, source становится `repeated_week`, timestamps создаются заново.
- Заметки, shopping sessions, отметки покупок, ручные товары, фактические цены и pending/error-состояния не копируются.
- Пустой исходный день никогда не очищает целевой день.
- Список покупок строится только по выбранным ужинам; базовые продукты добавляются только при включённом флаге.
- Одинаковые продукты объединяются по нормализованному названию; граммы/килограммы и миллилитры/литры суммируются.
- Статусы покупок и неотправленные записи сохраняются локально.

## Ограничения

- Несовместимые единицы объединяются текстом через `+`; точная стоимость может отсутствовать.
- Фильтр запрещённых продуктов основан на текстовом совпадении и не заменяет медицинскую проверку аллергенов.
- Клиентский API token виден пользователю и не является полноценной авторизацией.

## Открытые вопросы

- Нужна ли строгая защита от одновременного редактирования членами семьи.
- Какие единицы и правила пересчёта обязательны для достоверного бюджета.

## Shopping session

- Каждая закупка имеет стабильный `sessionId`, период плана, статус `active` / `completed` / `archived`, snapshot товаров и timestamp последнего изменения.
- Каждый товар имеет стабильный `itemId` и статус `to_buy` / `purchased` / `skipped`; ручной товар принадлежит только своей session.
- Новая закупка всегда сбрасывает товарные статусы. Глобальные legacy-отметки и manual items к новой session не применяются.
- Активная закупка не заменяется молча: пользователь продолжает её, завершает или архивирует перед созданием новой.
- Завершение с оставшимися `to_buy` требует явного подтверждения. Завершённая session доступна в Истории только для чтения и может быть скопирована без статусов.
- Быстрые отметки немедленно сохраняются в session-scoped local draft и отправляются полным snapshot с debounce через существующий pending-write механизм.
- Legacy sessions нормализуются на клиенте без массовой перезаписи Google Sheets.
