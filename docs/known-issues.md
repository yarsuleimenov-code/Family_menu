# Известные проблемы и backlog

## P1 — конкурентная запись может повредить данные блюда

- Модуль: `apps-script/CodeV2.gs`, `upsertDish_`.
- Условие: два запроса одновременно обновляют одно блюдо или структуру листов.
- Ожидается: атомарно сохраняется один согласованный вариант блюда и ингредиентов.
- Фактически: upsert, удаление ингредиентов и append выполняются без `LockService`; операции могут перемешаться.
- Причина: составная запись не защищена script/document lock.
- Рекомендация: lock с ограниченным ожиданием вокруг mutation handlers; после — concurrency integration test.
- Риск исправления: средний (таймауты/очередь записи). Покрытие: отсутствует.

## P2 — API-запрос не имеет timeout

- Модуль: `frontend/src/services/googleSheetsApi.ts`.
- Условие: Apps Script зависает или сеть не завершает запрос.
- Ожидается: контролируемая ошибка и доступный retry/fallback.
- Фактически: `fetch` может ждать неопределённо долго; UI остаётся в loading/saving.
- Причина: нет `AbortController`/deadline.
- Рекомендация: общий timeout и различимые ошибки timeout/network/API.
- Риск исправления: низкий. Покрытие: отсутствует.

## P2 — отсутствуют автоматизированные тесты бизнес-логики

- Модули: shopping builder, random dish, даты, pending writes, Apps Script mutations.
- Фактически: package scripts содержат только typecheck/build; регрессии правил ловятся вручную.
- Рекомендация: начать с unit-тестов чистых функций и контрактных fixtures API, затем один e2e smoke.

## P2 — нет строгой multi-user авторизации

- Клиентский token доступен в bundle; Apps Script может работать без token, если Script Property не задан.
- Решение требует бизнес-выбора модели доступа и не входит в техническую правку аудита.

## P3 — документация содержит исторические QA-снимки

- `docs/QA_REPORT.md` и `docs/DATA_QUALITY_AUDIT.md` описывают прежние live-проверки и endpoint.
- Их результаты нельзя считать текущими без повторного разрешённого live smoke.
